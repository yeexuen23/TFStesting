This Timing Reference Document contains important information and data on PSOS- 68020 performance. Data herein are useful in competitive evaluations against other real- time operating systems and executives, as well as in the design considerations and analysis of the user's application.

The sections which precede the timing data are important reading, as they discuss what is measured, how it is measured, and how to use the information.

28-1 Perspective

Timing is but one of several criteria for judging an operating system or kernel. Among others are ease-of-use, reliability and functionality. The relative significance of these factors depends on the application at hand.

However, detailed timing information can be very valuable in another respect - they reveal design flaws and imbalances. For example, timing formulae confirm that another executive's performance suffers severely for all but the smallest of applications, due to recurring, lengthy search loops and simplistic data structures. In other words, timing data reflect not just real- time performance, but fundamental design quality.

Many benchmarks quote only singular numbers for the simplest test cases. They mislead by being deliberately incomplete, and by idealizing system states and conditions. A detailed timing reference document, if available, should always be used for any meaningful evaluation.

28-2 The Key Parameters

Responsiveness and throughput are the two common metrics for benchmarking a real- time kernel. Of the two, responsiveness is usually the more important.

Responsiveness measures a system's ability to react to external events. A key parameter is Interrupt Response, or Latency. A real-time kernel must solve numerous, low-level race conditions on common data structures by disabling interrupts. The longest such interval during which interrupts are turned off in a kernel defines its Interrupt Latency, which directly affects the min;.mum interrupt separation and service time which can be safely designed into the user's application. Other relevant responsiveness indicators include the speed with which a kernel can synchronize a process to the interrupting event, and switch to it.

Throughput measures a kernel's transaction handling speed, and is given as execution times for system calls, particularly those which are heavily used in timing sensitive paths of an application. One often quoted number, the task switch, or rescheduling, time, measures the efficiency with which a kernel switches between processes. We have elected to simply include this in the timing for those system call cases which result in a process switch.

28-3 Apples and Oranges

Timing, like all specifications, are difficult to compare. Consider just the following caveats.

Most benchmarks publish only a single timing number for a system call, which invariably represents the best case, e.g. no rescheduling. Nor do such numbers reveal variable terms, such as search loops, which are conveniently dropped. Particularly for flawed designs which are heavily search-laden, such variables can easily overwhelm the published, best-case number.

A case in point is one executive which searches an all in one task chain after almost every system call, in order to dispatch a task, whether or not a task switch has occurred. Such searches are time-consuming, especially for designs running a good number of tasks, but are omitted from the published best-case numbers. Or worse, executives commonly suffer from interrupt latencies which too are variable functions, usually of the length of list structures. Again, best-case numbers may look acceptable. But reality could be sobering.

By contrast, PSOS is a highly balanced design. Key performance areas such as interrupt latency and preemptive system calls have fast, and fixed, timing. For all other system transactions, variable terms are strictly minimized, which will be evident from the furnished measurement data.

PSOS is very fast. Many systems only look fast.

We believe that benchmarks for real-time systems should document all significant cases and variables terms. Until then, it is still caveat emptor!

28-4 Measurement Methodology

Timing is measured by actual execution on known hardware. The PSOS version used is V4.1 for the 68020 processor.

28-4.1 Measurement Hardware

Execution and measurement are performed on a 68020 CPU clocked at 25Mhz, or 40 nanoseconds per clock cycle, with the cache turned off. Memory uses static RAM (i.e. accessed with zero wait-states).

28-4.2 Measurement Software

The Benchmark Suites are carefully designed. They verify hardware characteristics, and provide a known and reproducible measurement environment.

Each measurement involves 100 repetitions of a PSOS transaction. The time taken is then divided by 100 to obtain the benchmark time for a single instance of that PSOS transaction.

For example, to measure the preemptive SEND MESSAGE call, 100 identical processes are created with ascending priorities. Each executes long enough to Request a Message from an empty Exchange, and thus wait in a FIFO queue. The low-priority Root process starts the timer and the chain reaction by sending a message into the exchange, thereby dequeueing a waiting process, which preempts Root. This process in turn sends a message into the exchange, and is preempted by the next process. And so on. The last, 100th, process to execute simply stops the timer.

28-4.3 Measurement Path

Each measurement path involves a system call to PSOS. In some cases, PSOS returns control to the caller. In other cases, the PSOS dispatcher switches to run another process. Referring to the diagram below, the former cases measure the path from (A) to (B), inclusive of the TRAP to PSOS. The latter cases measure the path from (A) to (B)', which thus also includes a process context switch.

28-4.4 Other Considerations

The data presented are obtained through real execution. By contrast, many benchmarks are derived from formulae using published 68020 instruction timing data, and are therefore optimistic, since such 68020 timing data assumes optimal prefetch conditions. Our tests indicate that timing "computed" in this manner is 1. to 6% below actual execution time.

28-5 Data Interpretation

The following sections catalog the timing data for various PSOS System Services. Not all PSOS system calls are included. Some system services, Set - Time or Create/Delete Exchange for example, are typically not used in time-sensitive paths. The important system services to note are those commonly used for process communication, e.g. Message and Event services.

28-5.1 Measurement Cases

For most system services, the measured timing depends on the outcome of the transaction, which in turn depends on the temporal state of the system. Each different outcome is called a Measurement Case. For each system service in the catalog, all important measurement cases are included. However, it is simply not possible, nor educational, to list all measurement cases.

28-5.2 Measurement Data

Every measurement case has a timing number and a "Variant Condition". The timing number is the best-case timing for a single transaction. The variant condition shows the variable terms which may have to be added to that timing number, depending on the temporal state of the system, and parameter options.

28-6 Variant Conditions

28-6.1 (R) - Ready List Insertion

This condition results from inserting a process into an ordered ready list. To be added to the timing number is:

Variable Term = I * (R-1) microseconds

where R is the number of processes in the ready-to-run state which have higher priority than the process being inserted to the ready list.

It is important to note that at any given time, in a real-time application, the majority of processes are waiting for something. Therefore, the number of processes in the ready state is usually very small, so that the variable term (R) is insignificant.

28-6.2 (T) - Timeout Option

This condition occurs when a process waits for message, memory, or events, with the timeout option, thus requiring the insertion of the process into an ordered timeout list. To be added to the timing number is:

Variable Term = 2 * T microseconds

where T is the number of processes in the timeout state which will time out sooner than the process being inserted into the timeout list.

For transactions which wait without timeout, i.e. forever, this term is zero.

It is important to note that, in a typical application, only the few "things" which are external and may fail to materialize should be waited for with timeout. Therefore, the number of processes in the timeout state is usually very small, so that the variable term (T) is insignificant.

It is also interesting to note a common practice among many designs, wherein interrupts are disabled during the entire search and insertion to the timeout list. This adversely affects interrupt latency. By contrast, PSOS allows an interrupt-window between searching each element in the list.

28-6.3 (P) - Process Switching

The PSOS process-switch, or rescheduling, time is fixed, and appears as a 19 microsecond overhead over an identical transaction which does not result in a process switch. This time is already included in the appropriate measurement cases.

By contrast, many designs exhibit a variable rescheduling time. Worse, they often incur this overhead activity even in the absence of a process switch, i.e. when returning to the same process.

28-6.4 (M) - Memory Allocation

This condition occurs when the PSOS memory manager needs to search one or both of its free-segment lists in order to allocate a memory segment. To be added to the timing number is:

Variable Term = 2 * M microseconds

where M is the number of segments searched, but which were too small.

If the user always requests fixed-size segments from each memory region, then the variable term is zero, as searching is not necessary.

It is important to note that, in a typical real-time application, the allocation and de- allocation of memory segments are well within the user's control and anticipation. Even if variable-sized segments are used, the number of disjoint free segments should be quite small. This (M) term should not be significant.

28-6.5 (F) - Free Memory

This condition arises when a memory segment is returned, as PSOS searches the calling process's owned memory list. To be added to the timing number is:

Variable Term = 2 * F

where M is the number of memory segments allocated to, and owned by, the process subsequent to the segment now being returned.

Note that a typical process will not, at a given time, own many segments. Therefore, this term is insignificant. The benefits, on the other hand, are important, as this allows PSOS to perform automatic reclamation of segments when a process is deleted.

28-6.6 (A) - Active Chain Removal

This condition arises when a process is deleted and must be removed from an active chain. Since process deletion is generally not time critical, this active chain is singly- linked, so that removal requires searching. To be added to the timing number is:

Variable Term = .8 * A

where A is the number of active processes created subsequent to the process now being deleted.

This variable term could be significant if the number of active processes is large. However, process deletion is not typically time-critical.
