CHAPTER 18 Trigger Driver

18-1 Overview

The Trigger Driver is the software (SW) module that encapsulates the control of the trigger system for the JediPlus products. Logical Trigger Driver presents an Application Interface (API) that unifies and standardizes control over the logical blocks in the JediPlus trigger system. It accepts the trigger configuration parameters from the various trigger controls in the behavioral layer of the User Interface (UI) and maps it into the control parameters for the logical blocks in the trigger system. The primary purpose of this logical driver is to provide an abstraction layer interface between the UI controls in the behavioral layer of the UI and the product specific code that invokes the physical drivers to program the trigger HW. This standard API facilitates quick development of the follow on JediPlus products since it isolates the SW changes needed to support different Trigger HW implementations, down in the Physical Trigger Driver. The Physical Trigger driver accepts the trigger control parameters for the logical trigger blocks and maps them into the form needed to program the physical components implementing the logical blocks.

The Trigger Driver Context Diagram is a simplified view of the relationships between the Trigger Driver and the rest of the JediPlus SW System. Its shows the Logical Trigger Driver providing part of the hardware abstraction layer (HAL), between the UI and the Product Specific Trigger Driver. It shows the Product Specific Trigger Driver invoking the Physical Trigger Driver and the Physical Trigger Driver programming and monitoring the status of the trigger HW, that generated a system trigger to start each real-time acquisition.

18-1.1 The Logical Trigger Driver APIs

The Logical Trigger Driver in the JediPlus SW is composed of a data structure, called the Trigger Settings Template, and the set of functions that initialize, update, manipulate and propagate the information in the structure. The Trigger Template basically holds the state information for the Logical Trigger Driver. A simple view is that trigger settings information from the UI is passed into the driver and saved. This is used to generate additional control information for the various logical blocks in the Trigger System that is also saved. A mix of this UI and logic block information provides the basis for the API with the Physical Trigger Driver. The details of this API with the Physical Trigger Driver are presented in the next Section.

The UI uses the Trigger Mode concept to present the capabilities of the trigger system to the user. Basically the user is forced to first select an appropriate Trigger Mode and then set up the features pertinent to the selected mode. In general, there are separate control mechanisms in the UI behavioral layer for a feature if it appears in multiple modes. This allows us to retain control settings when modes are switched or promote settings between modes when they are switched, based on usability tests. The chosen behavior is encapsulated in the interaction between controls in the UI, hence it is of no real consequence to Logical Trigger Driver. The Logical Trigger Driver is free to perceive each mode as a separate trigger with its own independent set of control parameters.

The Trigger Template Structure

Like other Setting Templates in JediPlus, the Trigger Settings Template is conceptually made up of multiple sections, containing User Factors and Driver Factors. The TrigTemplate, defined in the file /jediPlus/include/dvrTrig.h, would typically contain distilled UI information called the Trigger User Factors. However, in this case the User Factor section is void so it only contains the trigger change register. The Trigger Driver Factors section, defined in the file jediPlus/hwDvr/include/dvrTrigUtil.h, consists of a change register, followed by a group of driver private parameters. The driver private parameters include some logical trigger state variables and private versions of several trigger calibration factors. The definitions are shown below:

Like the other Setting Templates, only a pointer to the DvrTrigTemplate, called "dvrTrigPtr", is instantiated in the JediPlus software. A pointer to the TrigTemplate called "trigPtr" is also instantiated. This scheme allows the UI to be knowledgeable about the change flags in the Trigger Template, without being cognizant about the DvrTrigTemplate and its Driver Factors or the product specific factors. This supports the notion that the Driver Factors could be different between product family members without impacting the UI.

The product specific trigger driver factors for muskrat are defined in muskTrigUtil.h. A pointer to the MuskTrigTemplate called muskTrigPtr is also defined in muskTrigUtil.h. There are no product specific trigger factors in puma so the PumaTrigTemplate looks just like the DvrTrigTemplate.

Trigger Driver Entry Point Functions

The entry point functions for the Logical Trigger Driver are defined in /jediPlus/include/dvrTrig,h. Notice they are somewhat different than those found in other Logical Drivers because they are Mode oriented. Basically, there is one entry point that handles the control changes for each trigger mode. When UI trigger controls change they do not generate and update distilled User Factors in the Trigger Template. Instead they simply update their piece of the UI triggerSettings structure and mark their "modeConfiguration" dependent control. This causes the effector routine for the "modeConfiguration" dependent to run later, where the call to Logical Trigger Driver entry point routine for that mode is executed. Typically a pointer to the section of the UI triggerSettings structure for that mode is passed in to the entry point routine, which saves a complete copy of the settings for that mode in the Driver Factor section. This means that the entry point function for each mode gets called when any change effecting the mode occurs with all the pertinent information. The entry point routines basically configure the mode completely from scratch. The entry point functions are:

18-1.2 The Logical Trigger Model

The logical model for the JediPlus Trigger System is shown in the figure below. The inputs to the logical model are logical versions of the trigger sources. These are the outputs of the trigger conditioning and trigger comparator blocks, which lie in the vertical paths for the analog channels, external trigger channels and timing pods. At this point the trigger inputs have already been compared to the trigger thresholds and appear as square waves with transitions at each signal / threshold crossing.

FIGURE 44. The Logical Model of the JediPlus Trigger System

The functional blocks in the logical trigger model include the simple Edge Trigger, the Pattern Duration Trigger, the State Trigger, the TV Trigger, the Sequence Trigger, the Serial Trigger, the Combination Trigger and the Trigger Type Selector. The first seven blocks are mode specific blocks, where the last block, the Trigger Type Selector, is used by all modes. Two blocks must be configured for each trigger mode, one mode specific and the Trigger Type Selector. A brief description of each block follows.

Edge Trigger Block

The Edge Trigger block consists of an edge detector that determines the source signal and polarity of the event that causes the trigger. It is used in the basic edge trigger mode.

Pattern Duration Block

The Pattern Duration Trigger block consists of a pattern detector that determines the logical combination of source signals and their logic levels (H,L,X) that can causes the trigger. Patterns that meet the pattern detector criteria are further qualified by a duration counter block that determines the upper and/or lower limits for the pattern duration that actually causes trigger.

This block is used in Pattern Duration Trigger mode where any combination of the analog channels and digital timing bits can be included in the pattern specification. It is also used in Pulse Width Trigger Mode where a single analog channel or digital bit must be used for the pattern specification.

State Trigger Block

The State Trigger Block consists of a pattern detector that enables the trigger when the logical combination of source signals and their logic levels is met, and an edge detector that causes the trigger when the specified signal transition occurs.

This block is used in the simple Pattern Trigger mode where any combination of the analog channels and digital timing bits can be included in the pattern qualifier specification, except for a signal that is specified as the edge source, (or state clock).

TV Trigger Block

The TV Trigger Block consists of two specialized trigger blocks that allow triggering on most serial video streams. The TV mode selector determines the video standard, field, line number, etc. that will cause the trigger when detected. The TV Source selector determines which analog channel is fed into the TV mode selector.

This block is used exclusively in the Advanced TV Trigger mode, but the Source Selector MUX must be disabled when it is not in use.

Sequential Trigger Block

The Sequential Trigger Block consists of two pattern detectors, two edge detectors, an event counter, and timeout counter and a sequencer block. The pattern and edge detectors are used for specifying trigger terms, which can be logically combined to create events. Occurrences of these events, or a timeout event, are then used to define the find, trigger and reset stages implemented by the sequence block.

This block is used in the advanced Sequence Trigger mode where any combination of the analog channels and digital timing bits can be included in the pattern or edge term specifications. The timeout event is only used in the reset sequence stage, currently.

Serial Trigger Block

The Serial Trigger Block consists of a serial data source selector, and serial clock source selector and a serial pattern selector. These can be used to specify phenomena in serial data streams to trigger on such as the beginning and end of frames, occurrence of special data or address packets, etc.

This block is used in the advance I2C trigger mode. Any analog channel or digital timing bit can be selected as the clock or data source. Trigger can be specified to occur on the I2C start condition, stop condition, or on frames containing reads/writes of specific 16 bit serial address/data values.

Combinational Trigger Block

The Combinational Trigger Block consists of two pattern detectors, and two edge detectors that can be combined logically using boolean algebra to generate triggers.

Currently this block is not used by any of the trigger modes implemented.

Trigger Type Selector Block

The Trigger Type Selector Block must be set up for each mode described above. It can be thought of as the block that maps the various HW trigger resources into the arrangement needed to implement a given mode. It consists of trigger resource blocks, trigger routing blocks, an event counter and a timer block. The resource blocks allow the various pattern and edge detectors to be combined logically to form the qualification and/or trigger events used by the modes above. The routing blocks accomplish several tasks, They route the inputs and outputs of the resource blocks, the timer, the event counter, the sequencer, etc. for the trigger mode in effect. They also disconnect the blocks not being utilized in a benign manner, so they don't effect triggering Finally, they make the final determination of the trigger event passed on the Acq System HW as the System Trigger.

Trigger Holdoff Block

The Trigger Holdoff Block controls the cycle time of the trigger system in all trigger modes. It consists of a Holdoff counter that must be counted down to zero before the trigger system is armed. The user can specify a trigger holdoff time that effectively gates off events that would otherwise cause a trigger.

Trigger Control Block

The Trigger Control Block provides holistic control over the complete Trigger System. It consists of the master trigger control flip flop, that completely enables or disables triggering. It also supports the notions of restarting (resetting) the trigger system and auto-triggering.

18-1.3 The Product Specific Trigger Driver APIs

The product Specific Trigger Driver takes care of any product specific functionality that is necessary and then calls in to the physical layer to set up the hardware. In Muskrat the product specific layer takes care of the hardware inversion in the front end (more about this in the next paragraph) and stores the trigger factors in the muskTrigTemplate. In Puma the product specific layer programs the high speed trigger path (Hamster).

In Muskrat the analog channels sometimes invert when their vertical gain is changed. This causes the polarity of the trigger signals to change on some vertical gain changes. Since the UI controls that detect the vertical channel inversion do not have any knowledge of the trigger mode in effect, they have no way to determine what adjustments need to be made to the trigger edge and pattern detectors being utilized. In this case the Logical Channel Driver, which has determined that a channel inversion has occurred, calls the special Product Specific Trigger entry point function below:

  • extern boolean setTrigChanPreampInvert(AnalogChanId chan, boolean hwInvert);

    This function is passed the channel Id for the channel whose polarity has changed and the current state of the hardware invert parameter. Using these, it updates the private chanInvert[] array in the muskrat trigger Template. Then it calls the appropriate entry point function for the current mode, described in the section above, based on the private trigType factor in the template. It passes the copy of the trigSettings for the current mode, saved earlier, so the entry point has all the information required to configure the mode completely.
