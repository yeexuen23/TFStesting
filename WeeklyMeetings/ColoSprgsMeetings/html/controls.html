CHAPTER 12 The Behavioral Layer and the Control Abstraction

12-1 Introduction

A large portion of the effort to implement an instrument is invested in the design and construction of the action routines that process UserRequests. For an oscilloscope, even one with just enough features, there are many different requests that can be applied to the system. The goal of the Behavioral Layer is to map UserRequests onto the underlying set of drivers provided by the Hardware Abstraction Layer, and to provide the appropriate state and status feedback in response.

There are some fairly well defined patterns of execution that recur within the action routines. A common example is bounds checking of user requested parameters so that only valid configurations are passed on to the drivers. Another pattern is the interdependency (coupling) of some settings and the algorithm to resolve setting conflicts. These recurring patterns are encapsulated by the control abstraction.

The design goals for this layer, aside from the functional and performance requirements, is to maximize reuse of the common logic and to expose all the unique policies of the behavioral layer for easy inspection. The reuse goal is accomplished by reducing the number of behaviors to be implemented and generalizing them into a few discrete classes. Most of the design can then be focused on the "marginal" behaviors, that is, focused on the differences in how various UserRequests are processed. This partioning supports the second goal of inspectability.

The design for the behavioral layer is based on the Denali control system. This is an "object-minded" approach that has been proven to satisfy the stated design goals. There are several important differences in the design of the JediPlus control system. These are:

  • No information about how the instrument's state is rendered is held within the control system.
    The formatting and appearence is determined in the user interface layer that lies above the controls.
  • The footprint of controls is reduced. This is in part due to removing the UI rendering features of the design. Also, improvements in data structure design have produced savings.
  • The overall performance of control system functions such as dependency management and execution deferal is improved through use of better data structures and algorithms.

More information about these design features are detailed later in this chapter.

The next section describes the evolution of the overall design concept of the control system. In the subsequent section is a discussion of the design of the JediPlus control system. The final section discusses the details of design and implementation of the control system.

12-2 The design premise of the control system

12-2.1 The control

The primary element of the control system is the control. The control is an abstraction of a portion of the underlying functionality of the instrument. It is analogous to the physical control elements that comprise the instrument's front panel.

Primary control types

Controls can be broadly classified into several basic styles of interaction:

  • Momentary actuation as in a simple normally open switch
  • Discrete "1 of N" and "M of N" selection as in a gang push button switch
  • Continuously variable selection as in a potentiometer
  • Alpha selection for labeling and describing

In addition to these, there is the notion of a meta-control that is not visable on the user interface of the instrument. It is used to achieve delayed computation or to tee several other controls into a common action. The application of the meta control is illustrated later in the chapter.

The control types are differentiated by having different kinds of settings. The momentary and meta have no data, the 1 of N is based on an enumeration, the M of N is based on a set, the continuous is built on an integral value and the alpha is based on a string. Beyond the underlying data type, the types of controls have different expressions of limits and algorithms for checking the setting against them.

Relationships between controls

Some controls are totally independent of other controls in the instrument. However, many controls are related because the setting of one determines the limits for another. For example, the setting of a channel's sensitivity is a factor in the limits and step size for the channel offset. So the chanOffsetCtrl is dependent on the chanScaleCtrl. This pattern of ordinate and sub-ordinate controls is the most common control relationship.

The basic control state machine

Despite differences in type or interdependencies, all controls share a common pattern of execution. The following figure depicts the state machine that executes a control's methods. The pattern is drawn in the form of a event flow diagram. This is more commonly known as a control flow diagram but that name might be a bit confusing in this context. The state machine logic is represented by the vertical bar (ProcessCtrlChange). The dashed lines are the events flowing in and out of the control and within it to select algorithms to execute. The circles are the data processing portions of the control, possibly implemented as functions or as portions of functions.

The figure shows the two main control methods: adjustLimits and checkSetting. These are shown as events coming into the control state machine from the outside. These methods are actually not part of the public API to controls.

FIGURE 20. Event flow diagram showing the generalized execution pattern for controls.

The logic of the control is quite simple for these methods. If the control is told to adjustLimits, a predefined routine is invoked. Many controls have limits that are constant. For example, the timebase vernier control can always be turned on and off. Therefore, the timebase vernier control does not need an adjustLimit routine. But in contrast, the delayed scale control (s/div) can only be set to a value less than or equal to the main scale control. So the delayed scale control needs to recompute its limits any time the main scale changes. The adjustLimit method for a control is invoked at power-on to insure the limits are initialized correctly and also when any other control changes that has a setting factoring in the control's limits.

When a new setting is applied to a control, the setting must be checked against limits. If the resultant setting is different than the previous setting, the new setting is put into effect. Any other controls that have limits dependent on the setting must be marked to have their limits adjusted and setting rechecked. Additionally, all views must be updated that are dependent on the control setting.

The momentary and meta controls are degenerate cases of this model as they have no associated setting. They do not have an adjustLimits routine nor a checkSetting routine.

12-5.4 Group Acquire

The acquire controls govern acquisition parameters other than timebase settings that pertain to all channels. In the JediPlus family, this is a small set of controls that includes the acquisition mode, average weighting factor and dither mode.

Additional controls exist for selecting the number of acquired points and sample rate, however these are not settable via the userInterface. In each case, these controls are defaulted into the "auto" mode in which the sample rate and acquired points are determined automatically based upon timebase settings. The sampleRateCtrl and pointsCtrl are connected to the userInterface only for querying the selected settings.
